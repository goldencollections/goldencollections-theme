## Later Reusable Proof Session

After the 7-SKU crown task, run one focused 60-90 minute proof capture session for broader deity jewellery fit and measurement evidence.

This does not mean one polished video. It means one practical owner session where Anil, the team, a phone camera, a measuring tape/ruler, representative idols/faces, and selected products are used to capture reusable proof assets for:

- crowns / mukut / kireedam
- deity short harams
- deity long harams
- Varalakshmi / Lakshmi / Amman setup items
- Balaji / Vishnu / Perumal setup items

The goal is to create first-hand visual proof that can be reviewed once, then reused across product pages, collection pages, fit-process pages, WhatsApp fit help, and UCP prompt optimization.

Use the session as an evidence capture block, not a content shoot.

## Exact Shot List

## How Anil Reviews Unclear Cases

Anil should review only the cases where judgement is required. The goal is to protect his time while keeping public claims accurate.

Unclear cases include:

- Product photo shows a ruler, but the actual measurement is hard to read.
- Product/category is unclear from the image or old title.
- Deity compatibility is uncertain: general deity vs Lakshmi/Amman vs Balaji/Vishnu vs other.
- Fit depends on idol posture, face width, crown style, hair/drape, or arch clearance.
- Product material, plating, included components, or SKU identity is not obvious from current records.
- A public caption would imply a claim like "fits", "for Balaji", "for Varalakshmi", "temple use", or "complete set".

Review method:

| Field | Anil decision |
| --- | --- |
| `owner_review_status` | `approved`, `needs correction`, `do not use`, or `needs better photo` |
| `safe_deity_group` | Example: `Varalakshmi/Lakshmi/Amman`, `Balaji/Vishnu/Perumal`, `general deity`, or `uncertain` |
| `safe_category` | crown, short haram, long haram, vaddanam, hastham/padam, thomala/vagamalai, etc. |
| `safe_fit_note` | One short buyer-facing note, with caveat if needed. |
| `do_not_claim` | Anything Codex must avoid saying publicly. |

If Anil is unsure, mark it as unsure. That is a useful decision. Codex should then use neutral language such as "check measurements and send photos before ordering" instead of inventing compatibility.

## Where Codex Takes Over

Codex takes over after assets and Anil review notes exist.

Codex can:

- Rename and organize the files by SKU, category, deity group, and proof type.
- Create contact sheets for fast review.
- Triage proof tiers:
  - Tier 3: measurement proof is visible and machine-readable enough for Codex to use confidently.
  - Tier 2: visual proof is present, but Codex should review contact sheets and ask Anil only if dimensions or deity fit are ambiguous.
  - Tier 1: metadata/product page exists, but no new public measurement claim should be added without Anil confirmation.
- Write safe alt text for approved measurement images.
- Draft product/collection fit notes using only visible proof and owner-approved facts.
- Add proof blocks to fit-process, Anil, measurement, crown, Varalakshmi, Balaji, and collection pages when approved.
- Prepare WhatsApp fit-help snippets and permission-safe proof captions.
- Build or refresh UCP prompt baselines and check whether prompt results improved.
- Prepare post-sprint KPI summaries that connect UCP prompt wins to add-to-cart, WhatsApp fit-help, orders, and revenue.

Codex should not:

- Publish customer, temple, or home setup proof without permission and approved wording.
- Claim official temple approval, priest approval, universal fit, ritual completeness, or material certainty unless the exact fact is confirmed.
- Enable WhatsApp live sending or automation as part of this workflow.

## Simple Post-Sprint KPI Tracking

Track each UCP sprint as a business loop, not only a search-quality loop.

### Definition Of A UCP Prompt Win

A UCP prompt win means:

1. The target prompt returns correct products or collections in the top 3.
2. The top result is in stock or still valuable as an SEO/product discovery asset.
3. The top result has a stronger proof layer after the sprint: measurement image, clearer alt text, safer fit copy, owner-approved deity/category mapping, or better disambiguation.
4. Wrong-type results are reduced or removed from top results.

The crown sprint baseline already showed strong top-result correctness for prompts such as `Varalakshmi crown for home pooja`, `deity mukut kireedam`, `goddess crown with size`, and `Balaji crown for idol`. The next question is whether those wins turn into buyer actions.

### Tracker Table

Use one row per prompt group after each sprint.

| Field | Meaning |
| --- | --- |
| `sprint_date` | Date the proof/UCP sprint shipped. |
| `prompt_group` | Example: crowns, short harams, long harams, Varalakshmi, Balaji. |
| `ucp_prompt` | Exact prompt tested. |
| `target_urls` | Product or collection URLs that should win. |
| `before_top3_correct` | Count before sprint. |
| `after_top3_correct` | Count after sprint. |
| `proof_change` | Measurement image, alt text, fit note, owner review, disambiguation, or none. |
| `add_to_cart_7d_before` | GA4 add-to-cart events for target URLs before sprint. |
| `add_to_cart_14d_after` | GA4 add-to-cart events for target URLs after sprint. |
| `whatsapp_fit_help_14d_after` | WhatsApp CTA clicks or manual inbound fit-help count for target prompt/product group. |
| `orders_14d_after` | Shopify/GA4 orders tied to target products or landing pages. |
| `revenue_14d_after` | GA4 purchase revenue or Shopify order revenue tied to target products/pages. |
| `owner_note` | What to do next: repeat, expand, pause, restock, or ask Anil. |

### Minimal KPI Set

Do not build a dashboard yet. Start with these five numbers per prompt group:

1. UCP top 3 correctness.
2. Number of top 3 results with visible measurement/proof.
3. Add-to-cart events on target product/collection URLs.
4. WhatsApp fit-help clicks or inbound messages for that category.
5. Orders and revenue for target URLs/products.

### Tracking Windows

Use simple windows first:

- Baseline: 7 days before the sprint.
- Early read: 7 days after the sprint.
- Main read: 14 days after the sprint.
- Seasonal read: 30 days after the sprint for Varalakshmi/Balaji/festival-sensitive prompts.

If traffic is low, do not over-read one week. Look for directional evidence:

- Did more shoppers add these products to cart?
- Did WhatsApp sizing questions become more specific and easier to answer?
- Did orders include products that won UCP prompts?
- Did wrong-category UCP results stop taking attention away from the intended products?

## Owner Decision Rules

- If a product has strong UCP visibility but weak proof, capture proof before adding more copy.
- If a product has proof but poor UCP visibility, let Codex adjust safe titles, alt text, collection routing, and disambiguation.
- If a product gets add-to-cart but no orders, check price, stock, shipping confidence, fit uncertainty, and WhatsApp questions.
- If a product gets WhatsApp questions but few add-to-carts, turn the repeated questions into product-page fit notes or image captions.
- If Anil cannot confidently approve a fit claim, keep the public copy conservative and invite buyers to send measurements/photos.
